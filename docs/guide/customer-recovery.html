Customer Recovery ​

Customer portal recovery surfaces help support teams and customers reconstruct payment activity without exposing private account data or relying on webhook delivery.

Payment history ​

Customer payment history should show one-time payments and recurring mandate charges in the same operational view when the gateway supports both. Rows should include support-safe references:

Field groupExamples
Lifecyclestatus, raw status, created time, updated time, confirmed time
Paymentamount, currency, description, merchant name, merchant reference
Recoverysession id, transfer reference, settlement rail reference, refund status
Route contextmerchant settlement account label, bank handle/name, masked IBAN, checkout channel name/environment

CSV exports should include the same support-safe route context as JSON rows:

  • merchant_account_label
  • merchant_account_bank_handle
  • merchant_account_bank_name
  • merchant_account_iban_masked
  • checkout_channel_name
  • checkout_channel_environment

Exports must not include full IBANs, raw account numbers, raw customer identifiers, raw API keys, webhook signing secrets, OAuth tokens, private keys, or unmasked webhook payload bodies.

Support-safe identifiers ​

Customer and operator recovery screens should use references that support teams can read back to a bank, merchant, or customer without exposing private identifiers. When a customer alias, bank customer reference, phone number, national ID, or portal email is needed for triage, prefer masked display fields:

IdentifierRecommended display
Phonelast four digits only, for example ***4567
National IDlast four characters only, for example ***1234
Portal emailmasked local part, for example o***r@example.com
Accountmasked IBAN or account alias only
Customer referencebank/customer reference only when it is already support-safe

Raw national IDs, raw phone numbers, raw portal emails, raw account numbers, passwords, reset tokens, OTPs, and authentication payload bodies should not appear in list rows, CSV exports, copied support summaries, screenshots, tickets, or logs. If an implementation needs exact values for matching, it should perform that matching server-side and return only masked values to the portal.

Customer alias route exports should include only customer-owned recovery fields such as alias, bank handle, active/inactive status, linked-account count, default masked account, default bank handle, and enrollment timestamp. They should not include full IBANs, raw account numbers, customer phone numbers, national IDs, credentials, OAuth tokens, private keys, signing secrets, or raw Identity Registry payload bodies.

Mandates and subscriptions ​

Mandate recovery should include:

  • mandate id, status, frequency, amount limit, currency, and merchant reference
  • consented, cancelled, created, and updated timestamps
  • payer bank handle and auth mode where available
  • support-safe settlement account and checkout channel route context
  • cancellation controls only where the backend supports customer cancellation

Historical mandate rows should keep their original route context after a merchant changes the default settlement account or reroutes a checkout channel.

Open Banking consents ​

Open Banking consent recovery should include:

  • consent id, TPP/client name, bank handle, scopes, and status
  • created, authorised, expiry, and revoked timestamps
  • revoke action where the consent state and backend integration allow it

Consent exports should avoid raw access tokens, refresh tokens, national IDs, private account numbers, and any bank credentials.

Bank customer recovery ​

Bank-support portals may expose a customer recovery view that resolves a bank customer reference or alias, then shows the customer's Open Banking consents and subscription mandates for that bank only.

Bank-support portals may also expose a registered customer report for the bank's OpenWave Identity population. This report should be useful for operations, enrollment support, and account-route recovery without becoming a raw identity export.

Recommended registered customer report fields:

Field groupSupport-safe fields
Aliasalias username, full alias, bank handle, active/inactive status
Customer referencebank customer reference when it is already support-safe
Contactmasked phone only
Enrollmentenrollment timestamp and account count
Default routedefault account id, account label, currency, masked IBAN

Registered customer reports should support search, active-only filtering, pagination, and capped CSV export. Search may match exact private values server-side when needed, but list rows and CSV exports should return only masked or support-safe values.

Neptune Astro exposes this implementation-specific bank portal report at GET /api/v1/bank/reports/customers. OpenWave Identity remains the source of truth for bank-linked aliases; its bank-scoped GET /identity/accounts endpoint accepts activeOnly, page, limit, and search so implementations can page at the source rather than loading every linked identity into the gateway.

Recommended bank customer recovery exports:

ExportSupport-safe fields
Registered customersalias username/full alias, bank handle, bank customer reference, masked phone, status, enrollment timestamp, account count, default account label/currency, masked default IBAN
Open Banking consentsconsent id, client id, TPP/app name, bank handle, scopes, status, created/authorised/expires/revoked timestamps
Subscription mandatesmandate id, status, merchant id/name/reference, settlement account id/label/bank context, checkout channel id/name/environment, payer bank handle, amount limit, currency, frequency, created/consented/cancelled timestamps

These exports should use the current portal filters, keep ownership bank-scoped, and apply the same bounded CSV behavior as payment and webhook reports. They should not expose raw phone numbers, national IDs, customer emails, OAuth tokens, token hashes, authorisation hashes, raw Open Banking payloads, full IBANs, raw account numbers, credentials, signing secrets, private keys, raw Identity Registry payload bodies, or raw mandate/payment payload bodies.

Finance and BNPL history ​

Finance recovery should connect customer-visible contracts/offers back to payment references:

  • contract id, offer id, product type, amount, currency, and status
  • payment session id and repayment mandate id where present
  • activated, cancelled, created, and updated timestamps
  • repayment schedule summaries without exposing sensitive account data

Readiness expectations ​

Customer recovery pages should keep loading, filter, export, and detail failures local to the affected section. A failed export or detail load should not clear unrelated history that already loaded successfully.

Recommended controls:

  • bounded pagination for JSON lists
  • capped CSV exports or internal paging
  • copyable support summaries with only support-safe fields
  • clear empty states and persistent warning panels for partial failures
  • no dependence on merchant webhook delivery for customer-visible recovery
