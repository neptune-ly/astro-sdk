Direct Merchant Integration ​

Astro supports direct merchant integration without a Neptune SDK. This path is for merchants that already have a backend team, want to call the REST APIs directly, and are approved for direct API access by operations.

Astro supports two direct-integration models:

  1. Standard direct API plus OpenWave-hosted checkout
  2. Trusted merchant-hosted checkout for explicitly approved merchants

If you are Ezone or another merchant already approved for trusted merchant-hosted checkout, start with Ezone Quick Start. That page gives the exact call sequence without SDK-oriented background material.

The default model used by most merchants, including the original Ezone setup, is:

text
Merchant backend -> Astro API -> hosted checkout -> signed webhook -> merchant order system

For a trusted merchant that is approved to keep the customer on the merchant site or app, Astro can also expose a separate merchant-hosted checkout API surface:

text
Merchant frontend -> merchant backend -> Astro merchant-hosted checkout APIs -> bank authorization via Astro orchestration -> signed webhook -> merchant order system

The security rule does not change:

  • merchant keys stay on the merchant backend
  • customers still complete authorization through Astro-owned bank orchestration
  • fulfilment waits for a signed final webhook such as payment.completed

Direct API vs SDK ​

Direct API access and SDK access are related but not equivalent:

  • Direct API means your backend can create sessions, query status, register webhook endpoints, and perform other server-to-server operations with a merchant key.
  • SDK means you may also use the official client or mobile helpers where Astro exposes them.
  • the standard direct API mode does not let the merchant replace hosted checkout with a merchant-owned bank authorization page
  • the trusted merchant-hosted mode is a separate approval and must be enabled explicitly per merchant
  • the merchant backend is the only supported caller to Astro for this mode; do not expose merchant API keys in browser or mobile client code
  • customer bank authentication still runs through Astro-orchestrated authorization, even when the customer remains inside the trusted merchant experience

Choose direct API when your team wants full server-side control and does not need client-side SDK helpers. Choose an SDK when you want maintained browser or mobile components, built-in webhook helpers, or a faster path for app-side integration.

Who should use this path ​

Use direct integration when all of the following are true:

  • the merchant has direct API access enabled
  • the merchant can safely store bearer keys on a backend service
  • the merchant can receive HTTPS webhooks
  • the merchant wants to call Astro over REST instead of using a maintained SDK

If the merchant needs browser components, mobile helpers, or a faster first integration, use one of the official SDKs instead.

Trusted merchant-hosted checkout ​

This capability is for a small set of trusted merchants only. It means the customer completes the payment flow inside the merchant website or app instead of being redirected to the OpenWave-hosted checkout page. The merchant does not open an OpenWave page in a redirect, popup, iframe, or embedded webview for this path.

For the compact endpoint-by-endpoint contract, see Trusted Merchant-Hosted Checkout.

Use this model only when all of the following are true:

  • Astro operations explicitly enabled merchant-hosted checkout for the merchant
  • the merchant already has direct API access and live approval where required
  • live merchants have at least one active managed merchant webhook endpoint for recovery and audit
  • the merchant can safely collect payer identifiers and bank step-up inputs inside its own PCI-like controlled experience
  • the merchant can implement stricter logging, masking, anti-fraud, and replay controls than a normal redirect integration

Important security boundaries:

  • merchant-hosted checkout is not a public replacement for hosted checkout
  • it must stay disabled for normal merchants
  • customer bank authorization is still validated by Astro and the bank path Astro orchestrates
  • final fulfilment still waits for the signed final webhook such as payment.completed

Trusted merchant-hosted endpoint family:

text
/api/v1/payments/sessions/{session_id}/merchant-checkout/*

Current flow:

  1. Merchant backend creates the payment session with the normal merchant key and checkout_surface=TRUSTED_MERCHANT_HOSTED_CHECKOUT. In Neptune Astro, this trusted session-create call must also include Idempotency-Key.
  2. Astro returns the trusted session_id plus the /merchant-checkout/* API root for that session.
  3. The merchant backend calls POST /api/v1/payments/sessions/{session_id}/merchant-checkout/access with the long-lived merchant key to mint a fresh short-lived merchant_checkout_session_token for that page load.
  4. The merchant frontend can call the merchant-hosted checkout endpoints directly with X-Session-Token while the customer stays on the merchant site.
  5. Astro accepts those browser calls only for trusted merchant-hosted sessions and only from merchant origins approved in Astro. Origin approval is enforced at the HTTP layer as well as inside the checkout authorization logic, so a merchant domain that is not explicitly approved will fail before checkout actions run.
  6. Merchant backend or frontend resolves payer input through the merchant-hosted checkout endpoints.
  7. Merchant frontend drives phone verification or typed payer selection where needed.
  8. Merchant frontend selects the bank authorization method exposed for that session.
  9. Merchant frontend submits the customer confirmation step through the trusted merchant-hosted confirm path.
  10. Merchant backend waits for signed webhook completion before fulfilment.

This model keeps the customer inside the merchant surface while preserving Astro ownership of session state, payer normalization, authorization state, and final payment outcome. The default remains hosted checkout even for a trusted merchant. Merchant-hosted checkout is used only when the merchant explicitly requests the trusted surface during session creation.

Trusted merchant-hosted auth model ​

  • Merchant backend calls POST /merchant-checkout/access with Authorization: Bearer mk_test_... or mk_live_....
  • Astro returns merchant_checkout_session_token.
  • Merchant frontend or backend proxy continues the flow with X-Session-Token: <merchant_checkout_session_token>.
  • The long-lived merchant bearer key stays backend-only.
  • The session token is short-lived, scoped to one payment session, and valid only for trusted merchant-hosted checkout.
  • Browser calls with X-Session-Token must come from an origin explicitly approved on the merchant profile.

Initiate contract for trusted merchant-hosted checkout ​

Set:

  • checkout_surface = "TRUSTED_MERCHANT_HOSTED_CHECKOUT"
  • Idempotency-Key: <stable value>

Live-only gates:

  • direct API access enabled
  • merchant-hosted checkout enabled
  • live enabled
  • at least one active managed merchant webhook endpoint

State machine ​

  1. PENDING
    • Resolve payer with alias, IBAN, or verified phone flow.
  2. OTP_SENT
    • Customer received bank OTP.
    • next_action = SUBMIT_OTP
  3. PUSH_SENT
    • Customer received bank push challenge.
    • next_action = POLL_PUSH_STATUS
  4. EXECUTING
    • Bank challenge passed and Astro is executing the payment.
    • next_action = POLL_PAYMENT_STATUS
  5. Terminal status:
    • COMPLETED
    • FAILED
    • EXPIRED
    • CANCELLED
    • RECONCILIATION_REQUIRED

Support-safe external status values remain the same as the standard merchant status API.

OTP and push specifics ​

  • Phone verification OTP:
    • Astro challenge TTL is 5 minutes.
    • Max failed verification attempts per challenge: 5.
    • Resend ceiling per session window: 3 sends.
    • Session/IP throttling is enforced before new challenges are created.
  • Bank payment OTP:
    • Merchant triggers or retriggers it through POST /select-auth with auth_mode = OTP.
    • Astro returns otp_expires_in_seconds when the bank reports it.
    • Invalid OTP returns INVALID_OTP.
    • Expired session returns SESSION_EXPIRED.
  • Push:
    • Merchant triggers it through POST /select-auth with auth_mode = PUSH.
    • Poll GET /push-status every 2 to 5 seconds.
    • Terminal push states are APPROVED, REJECTED, or EXPIRED.

Error model and idempotency ​

  • Merchant-safe error envelope:
    • code
    • message
    • request_id
    • optional details
  • Important codes in this flow include:
    • INVALID_OTP
    • SESSION_EXPIRED
    • SESSION_INVALID_STATE
    • PAYER_RESOLUTION_FAILED
    • IDEMPOTENCY_CONFLICT
    • bank-safe payment rejection codes such as INSUFFICIENT_FUNDS
  • Idempotency-Key is mandatory for trusted merchant-hosted session creation and recommended for all merchant write calls.

Completion and webhooks ​

  • payment.completed remains the authoritative completion event for trusted merchant-hosted checkout exactly as it does for hosted checkout.
  • Browser status and redirect-style signals are not the final fulfilment control.
  • Merchant systems should still verify and durably record the signed webhook before fulfilment.

Sandbox fixtures ​

Use the current Astro sandbox with:

  • approved sandbox merchant key
  • a valid sandbox alias or IBAN already provisioned in the connected test bank/identity environment
  • the standard sandbox webhook receiver path

The exact current sandbox customer fixtures and any bank-specific test OTP are bank-environment controlled and should be handed off by Neptune operations with the merchant enablement note, not hardcoded into public docs.

Compliance and browser/server split ​

  • POST /merchant-checkout/access is strictly backend-only.
  • All other /merchant-checkout/* calls may run either from the merchant backend proxy or directly from the approved merchant frontend with X-Session-Token.
  • Trusted merchants may collect payer identifiers and bank OTP on their own surface for this lane only.
  • Merchants must not store raw OTP values after submission, replay bank challenges outside Astro, or downgrade final fulfilment from signed webhook confirmation to browser-only state.

Environment model ​

Astro separates sandbox and live by credential and operator enablement. There is no request flag that turns a sandbox key into a live request.

EnvironmentTypical key prefixIntended use
Sandboxmk_test_...Integration build-out, test webhooks, non-production routing
Livemk_live_...Real customer traffic after readiness review

Rules:

  • sandbox and live are separate credentials with separate key rotation and separate webhook secrets
  • the credential prefix is authoritative for environment selection
  • a merchant may have sandbox enabled before live is approved
  • live traffic is enabled by Astro operations, not by a client-side toggle
  • merchants should use separate webhook endpoints, labels, logs, and recovery workflows for sandbox and live where possible

For operator onboarding, live enablement should require:

  • readiness review completed
  • default settlement account approved and active
  • at least one successful webhook test
  • support contact confirmed
  • merchant informed which live key or channel key to use

Minimal direct payment flow ​

1. Create the session from your backend ​

http
POST /api/v1/payments/initiate
Authorization: Bearer mk_test_...
Content-Type: application/json
Idempotency-Key: order_1042_create_session
json
{
  "amount": 860000,
  "currency": "LYD",
  "merchant_reference": "EZONE-1042",
  "description": "Ezone order 1042",
  "destination": {
    "type": "alias",
    "value": "merchant@andalus"
  },
  "redirect_url": "https://merchant.example.com/orders/EZONE-1042",
  "webhook_url": "https://merchant.example.com/openwave/webhooks"
}

When resolving an alias later in the flow, use a real enrolled NPT handle from the target environment, for example customer123@andalus. Do not use docs placeholders such as tellesy@andalus as if they were production fixtures.

The response returns payment_url, session_id, checkout_surface, authorized_checkout_surfaces, next_action, and, for trusted sessions only, merchant_checkout_api_root.

  • Standard direct integration: redirect the customer to the returned hosted page.
  • Trusted merchant-hosted integration: keep the session id and use the approved /merchant-checkout/* endpoints instead of redirecting to payment_url.
  • Even in trusted merchant-hosted mode, the customer stays on the merchant UX and the merchant frontend may call Astro directly only after the merchant backend exchanges the session into a short-lived merchant_checkout_session_token. Do not expose the long-lived merchant API key in browser or mobile client code.
  • Trusted merchant-hosted session creation is rejected if Idempotency-Key is missing.

2. Continue the customer flow on the approved surface ​

For the standard direct API model, the merchant backend should return the hosted URL to the frontend and then perform a normal redirect. The Astro checkout page is where the customer:

  • reviews the amount and merchant
  • chooses or confirms the payer route
  • authenticates with the bank
  • returns to the merchant redirect_url after the hosted step

For the trusted merchant-hosted model, the merchant stays inside its own frontend and uses the merchant-hosted checkout endpoints to:

  • resolve alias, IBAN, or verified phone payer input
  • select the allowed authorization path for the session
  • submit the bank authorization step through the trusted Astro API surface
  • poll push status where the bank flow is asynchronous

When the merchant frontend calls Astro directly, it must send:

  • X-Session-Token: <merchant_checkout_session_token>
  • the request from a merchant origin that Astro has explicitly allowed for that merchant

If the merchant wants to keep all Astro traffic server-to-server instead, that remains supported. The short-lived session token path is intended to make the browser experience cleaner without exposing the permanent merchant credential. The trusted browser path is never open-ended: it is gated by merchant approval, session-scoped token, explicit trusted checkout surface, and approved browser origin.

3. Wait for the signed webhook ​

Use the browser redirect only as a customer experience signal. Do not treat it as the final payment confirmation.

The merchant should fulfil only after:

  • the webhook signature is valid
  • the event is new or idempotently safe to replay
  • the event is a final state such as payment.completed
  • the event has been durably recorded or queued for reliable downstream processing before returning HTTP 2xx

Identifier handling: alias vs IBAN vs phone ​

When the merchant or customer provides a payer identifier, classify the value by the actual content.

Input shapeTreat as
LY...payer_iban
09... or +218...phone-based payer flow, when enabled
name@bank or nameNPT alias

For direct integrations:

  • do not send a Libyan IBAN through alias-labeled fields
  • do not label an IBAN as an NPT tag in callback or UI text
  • if the merchant UI uses a generic input, normalize it server-side before calling Astro

Recommended examples:

json
{
  "payer_alias": "tellesy@andalus"
}
json
{
  "payer_iban": "LY8302700000000000000000001"
}

Do not rely on free-form customer input reaching the API unchanged. Merchant systems should normalize and validate the identifier before submitting the request and should keep audit logs that show which typed field was used.

Webhook setup ​

Direct API merchants do not need an SDK to receive webhooks. They do need:

  • an HTTPS endpoint
  • raw-body signature verification
  • idempotent processing
  • delivery log recovery
  • separate operational handling for sandbox and live

Recommended endpoint management flow:

http
POST /api/v1/merchant/webhook-endpoints
Authorization: Bearer mk_test_...
Content-Type: application/json
json
{
  "label": "Ezone sandbox orders",
  "url": "https://merchant.example.com/openwave/webhooks",
  "event_types": ["payment.*", "refund.*", "mandate.*"]
}

The signing secret should be stored server-side and revealed only once at creation or rotation time.

Prefer managed webhook endpoints over the older single webhook_url compatibility field. Managed endpoints support event subscriptions, secret rotation, endpoint-specific logs, and clearer recovery. The request-level webhook_url on POST /api/v1/payments/initiate is only a per-session override; it does not replace your saved managed endpoint configuration.

Webhook receiver expectations:

  • verify X-OpenWave-Signature against the raw request body
  • use constant-time comparison
  • store event_id or equivalent idempotency marker before fulfilment
  • return HTTP 2xx only after the event is durably recorded
  • expect retries with backoff when your receiver is unavailable or returns non-2xx

Webhook recovery and reports ​

If the receiver is unavailable or the merchant loses an event, recovery should happen through the webhook delivery log and reports, not by asking operations to resend blind copies.

Useful API paths:

http
GET /api/v1/merchant/webhook-endpoints
GET /api/v1/webhooks?status=FAILED&limit=50
GET /api/v1/webhooks/health
GET /api/v1/merchant/reports/payments?format=csv

Recommended merchant recovery workflow:

  1. Inspect endpoint health and failed deliveries.
  2. Filter by date, event type, account, or checkout channel.
  3. Recover the payload details needed by the receiver.
  4. Retry or replay safely inside the merchant system.

Sandbox to live handoff ​

The handoff to live should be explicit.

Recommended operator checklist:

  1. Confirm the merchant can create sessions in sandbox.
  2. Confirm hosted checkout redirects correctly.
  3. Confirm a signed sandbox webhook is received, raw-body verified, and processed idempotently.
  4. Confirm the merchant understands the difference between redirect, non-final status, and final webhook state.
  5. Confirm the merchant has separate sandbox and live webhook URLs or an equivalent environment split.
  6. Enable live access for the merchant or selected live checkout channels.
  7. Deliver the live key or live channel key through the approved operator path.

Recommended merchant checklist:

  1. Replace mk_test_... with the approved mk_live_... or live channel key.
  2. Point live traffic to the live webhook endpoint and live webhook secret.
  3. Keep sandbox and live logs, secrets, and recovery procedures separate.
  4. Re-run a live test if operations policy allows it.
  5. Confirm fulfilment still waits for payment.completed, not only for redirect return.

Testing expectations ​

Before asking for live enablement, the merchant should be able to show:

  1. a sandbox session created with backend-only credentials
  2. a customer redirect into hosted checkout using the returned payment_url
  3. a valid X-OpenWave-Signature verification against the raw webhook body
  4. idempotent handling of duplicate deliveries
  5. status lookup or report-based recovery for a missed webhook
  6. internal logs that correlate merchant order reference, Astro session id, and webhook event id

Ezone-style onboarding wording ​

For merchants like Ezone, the operator message should be unambiguous:

  • no external SDK is required
  • direct API access is for backend calls; SDK access is optional and separate
  • the standard path uses Astro hosted checkout
  • the trusted merchant-hosted path keeps the customer fully on the merchant site or app and is available only for explicitly approved merchants
  • webhook integration is required for reliable automation and recommended even when polling exists
  • sandbox and live keys, secrets, and endpoints stay separate
  • live access is available only after Astro operations enable it

That wording avoids implying that every merchant must build its own checkout SDK or bank-auth surface, while still making the trusted merchant-hosted lane clear for merchants that Neptune explicitly approves.
