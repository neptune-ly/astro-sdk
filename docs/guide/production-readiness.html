Production readiness

Ship Astro integrations with clear sandbox/live boundaries and trusted branding.

Astro SDKs keep merchant keys server-side, use hosted authorization surfaces for customer decisions, and expose branding metadata so checkout, QR, NFC, Open Banking, and mandate flows identify the merchant and operating environment clearly.

Environment separationUse separate sandbox and live keys, webhook secrets, redirect origins, and settlement configuration. Never switch a live payment by a browser flag or customer input.
Trusted brandingShow merchant display names, approved logos, support contacts, and bank/operator identity from server-returned metadata, not frontend overrides.
Final-state fulfilmentFulfil only after a signed final webhook or trusted server status. Browser callbacks and settlement-pending states are not proof of completion.

Merchant keyUse sandbox credentials for tests and demos.Use live credentials only from the production backend.

Do not put live merchant keys, webhook secrets, bank credentials, OTPs, PINs, passcodes, or push approvals in browser or mobile merchant code.

Branding metadata

Astro-hosted checkout and SDK sessions may include merchant, bank, TPP, and gateway branding. Render these values as display metadata only; do not trust frontend-supplied branding for account routing, authorisation, fulfilment, or settlement decisions.

{
  "environment": "SANDBOX",
  "merchant_branding": {
    "display_name": "Ezone",
    "logo_url": "https://astro.example.com/api/v1/assets/branding/merchant_ezone.png",
    "brand_color": "#07315F",
    "support_email": "support@example.com",
    "website": "https://merchant.example"
  }
}

SDK handling

Backend SDKs create the session or presentment. Frontend SDKs open the returned hosted URL or scoped session and allow Astro to render the approved merchant and operator identity. Merchant pages may show the same approved logo beside a “Pay with Astro” button, but they must not collect the bank decision themselves.

Launch checklist

  • Use server-side session creation with idempotency keys and a clear order reference.
  • Configure allowed redirect origins per environment.
  • Upload and review merchant logos, display names, brand colors, support email, and website.
  • Verify signed webhooks for payment, mandate, Open Banking, finance, refund, and failure events used by the integration.
  • Test QR and NFC presentment claim flows through the returned hosted or SDK-controlled authorization surface.
  • Confirm production callback URLs, webhook retry handling, reconciliation, and operational contacts before live credentials are issued.
