On this page

Merchant Postman Project ​

Use this project when a merchant team wants a runnable Postman workspace instead of piecing together raw OpenAPI files. It covers the merchant surfaces that matter in Astro:

  • hosted checkout via Merchant API
  • trusted merchant-hosted checkout
  • webhook setup and replay
  • reports and readiness
  • recurring mandates
  • presented payments
  • open banking
  • credit and finance

This is for merchant backends, QA, operations, and partner enablement. It is not a browser-side integration artifact.

Downloads ​

  • Merchant suite collection
  • Sandbox environment
  • Live environment

What this collection includes ​

The collection is organized as a real runbook:

  1. 01 Public and Merchant Baseline
  2. 02 Hosted Checkout via Merchant API
  3. 03 Trusted Merchant-Hosted Checkout
  4. 04 Merchant Webhooks and Recovery
  5. 05 Merchant Reports and Readiness
  6. 06 Recurring Mandates
  7. 07 Presented Payments
  8. 08 Open Banking
  9. 09 Credit and Finance

Requests capture IDs and tokens into collection variables so the next step can run without manual copy/paste.

Recommended run order ​

Standard merchant checkout ​

  1. Import the collection.
  2. Import the sandbox or live environment.
  3. Fill in merchant_api_key, redirect_url, cancel_url, and merchant_webhook_url.
  4. Run 02 Hosted Checkout via Merchant API -> Create Hosted Payment Session.
  5. Open the returned payment_url if you want the OpenWave-hosted customer flow.
  6. Use Get Payment Session, webhook logs, and reports to verify completion.

Trusted merchant-hosted checkout ​

This path is for approved merchants only. It keeps the customer fully inside the merchant website or app.

  1. Use a merchant key with trusted checkout enabled.
  2. Run 03 Trusted Merchant-Hosted Checkout -> Create Trusted Merchant-Hosted Session.
  3. Run Exchange Merchant-Hosted Access Token.
  4. Continue with one payer path:
    • Resolve Payer by Alias
    • Resolve Payer by IBAN
    • phone flow: Phone Start, Phone Verify, Phone Select
  5. Run Select Auth Method.
  6. If the bank returned OTP, run Confirm OTP.
  7. If the bank returned push, poll Get Push Status.
  8. Always finish with webhook-backed verification and Get Trusted Checkout Status.

Open banking ​

  1. Fill in tpp_client_id and tpp_client_secret.
  2. Use a real PKCE challenge and verifier pair.
  3. Run the 08 Open Banking folder in order.
  4. After token exchange, the account and payment order requests use the captured bearer token automatically.

Variables you must set ​

VariableRequired forNotes
merchant_api_keyAll merchant payment, webhook, report, mandate, presentment, and finance callsUse mk_test_... in sandbox and mk_live_... in live.
tpp_client_id / tpp_client_secretOpen banking consent and token exchangeServer-to-server only.

merchant_webhook_urlWebhook endpoint setupUse an HTTPS receiver you control.

otp_codeOTP confirmation requestsLeave blank in live until the customer receives it.

Security boundary ​

Keep the split clear:

  • merchant API keys stay on the merchant backend
  • /merchant-checkout/access is backend-only
  • X-Session-Token is short-lived and scoped to one trusted checkout session
  • fulfilment must wait for the signed final webhook such as payment.completed
  • do not store raw OTP values after submission

Live testing notes ​

Live testing is not just a different URL. It requires:

  • live merchant key or live channel key
  • live enablement on the merchant profile
  • approved settlement routing
  • active webhook endpoint
  • trusted merchant-hosted approval if that flow is being used

For live alias, IBAN, and phone tests, replace the placeholder values in the live environment with real merchant-approved production data. Public docs do not ship live customer fixtures.
