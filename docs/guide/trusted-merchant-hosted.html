Trusted Merchant-Hosted Checkout ​

Trusted merchant-hosted checkout lets an approved merchant keep the customer fully inside the merchant website or app. There is no OpenWave redirect, popup, iframe, or embedded page for this path.

This is not the default direct integration model. The default model is Hosted Checkout via Merchant API, where the merchant backend creates a session and the customer completes payment in OpenWave-hosted checkout.

When this is allowed ​

Astro must explicitly enable trusted merchant-hosted checkout for the merchant. Live traffic also requires:

  • live environment enabled
  • direct API access enabled
  • a live merchant key or live channel key
  • at least one active managed webhook endpoint
  • approved browser origins for frontend calls, unless the merchant proxies every checkout call through its backend

All endpoints after /access are under:

text
/api/v1/payments/sessions/{session_id}/merchant-checkout

Required headers ​

Session creation and access exchange use the merchant key on the backend:

http
Authorization: Bearer mk_live_...
Idempotency-Key: merchant-order-reference

Trusted checkout browser calls use the session token:

http
Origin: https://merchant.example
X-Session-Token: <merchant_checkout_session_token>

The Origin must match an allowed merchant origin configured in Astro.

Frontend and backend split ​

The merchant backend may:

  • store merchant API keys
  • create payment sessions
  • exchange trusted sessions into short-lived checkout tokens
  • perform server-side status checks
  • receive and verify signed webhooks

The merchant frontend may:

  • collect payer alias, IBAN, or phone number
  • collect bank OTP only for this trusted lane
  • call trusted merchant-hosted checkout endpoints with X-Session-Token
  • show masked account choices from verified phone lookup
  • show progress while waiting for webhook-backed fulfilment

The merchant frontend must not:

  • store long-lived merchant keys
  • store OTP values after submission
  • simulate bank approval locally
  • fulfil an order from browser state alone

Status and next action ​

Trusted checkout responses include next_action and may include next_action_details. Use those fields to drive the merchant UI instead of guessing state transitions.

Common actions:

next_actionMerchant UI behavior
EXCHANGE_MERCHANT_CHECKOUT_ACCESSBackend must call /merchant-checkout/access.
RESOLVE_PAYERAsk customer for alias, IBAN, or phone.
VERIFY_PHONEAsk for phone OTP and show masked choices after verification.
SELECT_AUTHShow allowed bank authorization methods.
SUBMIT_OTPAsk for the bank OTP.
POLL_PUSH_STATUSPoll /push-status.
POLL_PAYMENT_STATUSPoll /status or wait for webhook.
PAYMENT_COMPLETEDWait for signed webhook before fulfilment if not already recorded.

Error handling ​

Show merchant-safe errors only. Do not expose raw bank or CBS text to customers.

CodeMeaning
INVALID_OTPThe OTP was not accepted.
SESSION_EXPIREDThe checkout session expired.
SESSION_INVALID_STATEThe requested step is not valid for the current session.
PAYER_RESOLUTION_FAILEDThe payer could not be resolved.
IDEMPOTENCY_CONFLICTThe idempotency key was reused with different request data.
BANK_UNAVAILABLEThe bank is temporarily unavailable.

Fulfilment rule ​

payment.completed remains the authoritative completion signal. Status polling helps the customer experience, but fulfilment must be backed by a signed webhook or a server-side final status lookup that the merchant records durably.
