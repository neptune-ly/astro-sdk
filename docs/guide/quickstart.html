Quick Start ​

This guide shows the shortest secure path to a production-style Astro integration. The key rule is simple: create intent on your backend, authorize the customer in Astro-hosted UI or an SDK-controlled webview, then fulfil only after a signed webhook.

1. Choose your integration shape ​

App typeUseWhy
Plain websiteWeb Drop-inAdds a hosted checkout modal with one script tag.
React storeReact SDKComponents and hooks around the same hosted checkout model.
Node backendTypeScript SDKCreate sessions, mandates, Open Banking consents, and verify webhooks.
Spring/JVM backendKotlin SDKSame server-side responsibilities for JVM systems.
Mobile appFlutter or Swift SDKLaunch secure hosted checkout or consent in a controlled webview.

2. Create a payment session on your backend ​

Never put a merchant API key in a browser or mobile app.

ts
import { AstroClient } from '@neptune.fintech/astro-sdk'

const astro = new AstroClient({
  baseUrl: process.env.ASTRO_BASE_URL!,
  apiKey: process.env.ASTRO_MERCHANT_KEY!,
})

const session = await astro.payments.createSession({
  amount: 860000,
  currency: 'LYD',
  orderReference: 'NS-10042',
  description: 'Neptune Store order',
  callbackUrl: 'https://merchant.example.com/webhooks/astro',
  returnUrl: 'https://merchant.example.com/order/NS-10042',
})

return { checkoutUrl: session.checkoutUrl, sessionId: session.id }

3. Open hosted checkout ​

Use the Web Drop-in, React SDK, Flutter SDK, or Swift SDK to open the checkout URL returned by the backend. The customer enters NPT alias or IBAN and authorizes with bank OTP or push inside the Astro surface.

html
<button id="pay">Pay with bank</button>
<script type="module">
  import { AstroCheckout } from 'https://cdn.example.com/astro-web.js'

  document.querySelector('#pay').addEventListener('click', async () => {
    const { checkoutUrl } = await fetch('/api/create-payment').then(r => r.json())
    await AstroCheckout.open({ checkoutUrl })
  })
</script>

4. Verify the webhook before fulfilment ​

ts
app.post('/webhooks/astro', express.raw({ type: 'application/json' }), async (req, res) => {
  const signature = req.header('X-OpenWave-Signature')
  const event = astro.webhooks.verify(req.body, signature, process.env.ASTRO_WEBHOOK_SECRET!)

  if (event.type === 'payment.completed') {
    await orders.markPaid(event.data.orderReference, event.id)
  }

  res.sendStatus(200)
})

5. Add recurring payments ​

Create a mandate from your backend, show the customer the amount, frequency, merchant, and expiry inside hosted approval, then charge only while the mandate is active and within approved limits.

ts
const mandate = await astro.mandates.create({
  customerReference: 'cust_1001',
  amount: 10000,
  currency: 'LYD',
  frequency: 'MONTHLY',
  description: 'Neptune Store Care+ monthly plan',
  returnUrl: 'https://merchant.example.com/subscriptions',
})

6. Add Open Banking ​

Open Banking uses OAuth-style consent. Your backend creates a consent request with explicit scopes and PKCE; the customer sees those scopes in the hosted consent screen; your backend exchanges the returned code for a scoped token.

ts
const consent = await astro.openBanking.createConsent({
  scopes: ['accounts:read', 'balances:read', 'transactions:read'],
  purpose: 'Show your bank accounts inside Neptune Store checkout',
  redirectUri: 'https://merchant.example.com/ob/callback',
  codeChallenge,
  codeChallengeMethod: 'S256',
})

7. Add financed checkout ​

When your Astro deployment exposes Credit & Finance, it follows the same security rule: your backend creates the finance intent, the customer reviews assessment purpose and offer terms in hosted or SDK UI, and fulfilment waits for final signed payment state.

ts
const assessment = await astro.finance.assessments.create({
  consentId,
  purpose: 'BNPL',
  requestedAmount: 860000,
  currency: 'LYD',
  tenor: { unit: 'MONTH', value: 6 },
  dataWindow: { from: '2025-11-01', to: '2026-05-01' },
  selectedAccountIds: ['acc_01JCR5YSS1QYZYKJ6W9AC4S5C8'],
})

const offer = await astro.finance.offers.create({
  productType: 'BNPL_INSTALLMENT',
  assessmentId: assessment.assessmentId,
  amount: 860000,
  currency: 'LYD',
  tenor: { unit: 'MONTH', value: 6 },
  merchant: {
    merchantId: 'mrc_01JCR6DP3A3J3YCJAWM5KWBYG2',
    displayName: 'Neptune Store',
    reference: 'NS-10042',
  },
})

Redirect the customer to offer.acceptUrl. Do not show raw assessment details to the merchant order system.

Go-live checklist ​

  • Merchant keys are stored only on your backend.
  • Hosted checkout or SDK webview is used for payments, mandates, and Open Banking consent.
  • Webhook signature verification is mandatory before order fulfilment.
  • Idempotency keys are used for session, mandate, refund, and payment-order creation.
  • Amounts are minor units.
  • Error messages shown to customers are clear, but internal bank or CBS details stay in logs.
  • Sandbox and production keys are separated.
  • Credit and finance consents are purpose-specific and show amount, tenor, data window, and repayment disclosures.
