Testing & Sandbox ​

Use the sandbox to prove each flow before going live. Keep sandbox keys, webhook secrets, and customer test data separate from production.

Test matrix ​

ScenarioExpected result
One-time payment by NPT aliasAlias resolves, customer authorizes, payment completes, webhook arrives.
One-time payment by IBANIBAN resolves to debtor bank, customer authorizes, payment completes.
Wrong OTPSession stays recoverable and returns a clear authorization error.
Expired OTPCustomer can request a new authorization challenge.
Push authorization approvedSession moves from AUTH_REQUIRED to processing/completed.
Push authorization rejectedSession fails with a customer-safe message.
Webhook receiver downAstro retries and exposes delivery history.
Recurring mandate approvalCustomer sees amount/frequency and mandate becomes active after SCA.
Mandate chargeCharge succeeds only while mandate is active and within approved limits.
Open Banking consentCustomer sees scopes, approves, backend exchanges code, scoped APIs work.
Consent revocationExisting tokens stop working after revocation.
Credit assessment consentCustomer sees finance purpose, amount, tenor, accounts, and credit scopes.
BNPL offer acceptanceCustomer reviews installments and disclosure, financier pays merchant, signed webhook arrives.
Murabaha offer acceptanceCustomer sees asset, cash price, profit, total sale price, and schedule before acceptance.
Finance declineMerchant receives safe reason category and correlation ID, not raw bank facts.

Webhook receiver ​

For local testing, expose your webhook receiver with a tunnel and use the public HTTPS URL as callbackUrl.

bash
ngrok http 4000

Then start a local receiver:

ts
app.post('/webhook', express.raw({ type: 'application/json' }), (req, res) => {
  const event = astro.webhooks.verify(
    req.body,
    req.header('X-OpenWave-Signature'),
    process.env.ASTRO_WEBHOOK_SECRET!
  )

  console.log(event.type, event.id)
  res.sendStatus(200)
})

Customer-facing error tests ​

Check that the UI never shows internal CBS or bank integration details. Customers should see clear recovery actions:

Technical causeCustomer-safe message
ALIAS_NOT_FOUNDWe could not find that NPT alias. Check the spelling or use IBAN.
PAYER_ACCOUNT_DEBIT_BLOCKEDThis account cannot be used for payment. Choose another account or contact the bank.
OTP_INVALIDThe code was not correct. Try again or request a new code.
BANK_UNAVAILABLEThe bank is temporarily unavailable. Try again shortly.
CONSENT_REVOKEDPermission was revoked. Start a new finance consent to continue.
FINANCE_OFFER_EXPIREDThis finance offer expired. Request a new offer.
FINANCE_POLICY_DECLINEDThis finance option is not available for the selected amount or tenor.

Credit & Finance test path ​

mermaid
flowchart LR
  Consent["Finance consent"] --> Assessment["Assessment completed"]
  Assessment --> Offer["Offer created"]
  Offer --> Acceptance["Hosted / SDK acceptance"]
  Acceptance --> Payment["Financier payment"]
  Payment --> Webhook["Signed merchant webhook"]
  Webhook --> Contract["Customer contract + schedule"]

Validate both positive and negative paths:

  • Consent rejected blocks assessment creation.
  • Revoked consent blocks assessment refresh.
  • Offer acceptance is refused from merchant-owned forms.
  • Merchant fulfilment waits for signed payment.completed.
  • Customer portal shows contract and repayment schedule.

Go-live evidence ​

Before production, capture:

  • Successful checkout by NPT alias.
  • Successful checkout by IBAN.
  • Failed authorization and recovery.
  • Signed webhook verification.
  • Recurring mandate approval, charge, and cancellation.
  • Open Banking consent, account read, and revocation.
  • Credit & Finance consent, assessment, offer, acceptance, payment, and repayment schedule.
  • Dashboard audit trail for payment, mandate, consent, and webhook delivery.
