Presented Payments ​

Neptune. Astro follows the OpenWave presented-payments model for merchant-presented QR, NFC, and app-handoff flows. The merchant or payment acceptor creates the presentment; the customer pays by scanning or tapping it.

The short version ​

QR or NFC starts the flow. Astro still owns the secure authorization step and then reuses the normal payment or mandate lifecycle.

What Astro can do in this model ​

CapabilityWhat Astro does
Merchant-presented QRCreates presentment, returns QR payload, opens hosted checkout after claim
Merchant-presented NFCCreates NDEF-compatible payload and resumes secure authorization after tap
Presented mandate approvalCreates a pending recurring mandate after claim and returns the hosted mandate consent URL

API surface ​

Astro exposes the OpenWave presented-payment lifecycle under the standard paths:

MethodPathAuthPurpose
GET/api/v1/capabilitiesPublicDiscover whether QR, NFC, app handoff, merchant-presented, and mandate-presented flows are enabled
POST/api/v1/presentmentsMerchant keyCreate a replay-safe QR or NFC presentment
GET/api/v1/presentmentsMerchant keyList merchant presentments
GET/api/v1/presentments/{id}Merchant keyRead merchant presentment detail
POST/api/v1/presentments/{id}/claimPublic claim tokenClaim a presentment and create the underlying secure flow
POST/api/v1/presentments/{id}/cancelMerchant keyCancel an unclaimed presentment
GET/api/v1/presentments/{id}/statusPublicPoll presentment state

Creation returns an OpenWave URI such as:

text
openwave://presentments/{presentment_id}?token={claim_token}&channel=QR&intent=ONE_TIME_PAYMENT&operator={operator_id}

The claim_token is only returned on creation and is stored hashed by Astro. After claim, a one-time presentment moves to PAYMENT_SESSION_CREATED and Astro returns payment_url plus auth_surface. A mandate presentment moves to MANDATE_SESSION_CREATED and returns mandate_consent_url plus auth_surface.

auth_surface is the contract: it is the secure hosted checkout, hosted mandate consent page, SDK sheet, or future bank-app handoff where customer authorization happens. Claiming a QR or NFC presentment never authorizes payment by itself.

For QR, Astro also emits an EMV/NUMO-compatible qr_payload.value when the channel is QR. The default OpenWave template is tag 26 with 26.00 = LY.OPENWAVE, 26.01 = presentment_id, 26.02 = claim_token, 26.03 = QR, and 26.04 = P for payment or M for mandate approval. Optional operator metadata uses tag 50 with 50.00 = LY.OPENWAVE.OP and 50.01 = operator_id. Existing NUMO/LYPay QR codes keep the old route; scanners enter the Astro/OpenWave path only when they find the LY.OPENWAVE GUI.

For NFC, Astro emits an NDEF URI-style payload, preferably an HTTPS universal/app link, that resolves to the same claim flow. A tap starts the presentment claim; it does not authorize the payment.

For production retries, send Idempotency-Key on POST /api/v1/presentments. Astro stores a merchant-scoped request fingerprint. Retrying the same key with the same body returns the existing presentment and QR/NFC payload; reusing the key with a different body returns 409 IDEMPOTENCY_CONFLICT.

ts
const presentment = await astro.presentments.create({
  channel: 'QR',
  mode: 'MERCHANT_PRESENTED',
  intent: 'ONE_TIME_PAYMENT',
  amount_mode: 'FIXED',
  amount: 125_000,
  currency: 'LYD',
  description: 'In-store checkout',
  merchant_reference: 'pos-10045',
}, {
  idempotencyKey: 'pos-10045-presentment',
})

What Astro does not do ​

  • It does not treat QR or NFC as final authorization.
  • It does not let merchants collect bank OTP, PIN, passcode, push-approval results, or bank credentials directly.
  • It does not collapse payment consent and Open Banking consent into one unsafe shortcut.

Presented recurring approval ​

Use this when the customer starts a subscription from a QR or NFC handoff.

The customer must still see:

  • merchant name
  • recurring amount or max amount
  • frequency
  • expiry or duration
  • cancellation rights

Only after that can Astro collect bank OTP or push approval.

Mandate approval presentments create a pending recurring mandate and return mandate_consent_url plus auth_surface. The QR or NFC tap is only the handoff; the customer must still review merchant name, amount limit, frequency, payer account, and complete OTP or push approval inside the hosted OpenWave mandate page before the mandate becomes active.

Security rules ​

RuleWhy
Presentments must expirePrevent stale reuse
Claim must be replay-safePrevent duplicate or spoofed payment start
Hosted or SDK-controlled SCA is mandatoryProtect bank secrets
Merchant fulfilment waits for signed final statePrevent early fulfilment on an incomplete flow
Claim token is stored hashedPrevent leaked database state from becoming a usable QR/NFC claim
Claim changes state oncePrevent double-spend and duplicate session creation

Direct bank or wallet implementations ​

OpenWave allows banks and wallets to implement the same presented-payment channel directly. Astro is one gateway implementation of that standard. If Astro interoperates with those direct implementations later, the capability discovery and claim lifecycle should remain identical.
