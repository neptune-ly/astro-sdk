Ezone Quick Start ​

This is the shortest path for Ezone-style trusted merchant-hosted checkout. The customer stays inside the merchant website or app. The merchant backend still owns the long-lived key, and Astro still owns payer resolution, bank authorization, execution state, and signed webhook delivery.

Use this page when Astro Operations confirmed that the merchant has:

  • direct API access enabled
  • trusted merchant-hosted checkout enabled
  • sandbox or live key issued through the approved credential handoff
  • at least one managed webhook endpoint for live recovery
  • approved browser origins for frontend calls to Astro

1. Create the payment session ​

Call Astro from the merchant backend only. Never put mk_test_... or mk_live_... keys in browser or mobile code.

http
POST /api/v1/payments/initiate
Authorization: Bearer mk_test_...
Idempotency-Key: EZONE-ORDER-1042
Content-Type: application/json
json
{
  "checkout_surface": "TRUSTED_MERCHANT_HOSTED_CHECKOUT",
  "amount": 860000,
  "currency": "LYD",
  "merchant_reference": "EZONE-1042",
  "description": "Ezone order 1042",
  "redirect_url": "https://ezone.ly/orders/EZONE-1042"
}

The response includes session_id, checkout_surface, authorized_checkout_surfaces, next_action, next_action_details, and merchant_checkout_api_root.

2. Exchange for a checkout session token ​

The merchant backend exchanges the trusted session into a short-lived checkout token.

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/access
Authorization: Bearer mk_test_...

Use the returned merchant_checkout_session_token as:

http
X-Session-Token: <merchant_checkout_session_token>

This token is scoped to one payment session and one trusted checkout flow. It is not a merchant API key.

3. Resolve the payer ​

From the approved merchant frontend or from the merchant backend proxy, call one payer path.

Alias or IBAN ​

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/resolve-payer
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>
Content-Type: application/json
json
{
  "payer_identifier": "<registered_npt_handle>@<bank_handle>"
}

Identifier rules:

Customer inputSend or classify as
LY...IBAN
09... or +218...phone flow
name or name@bankNPT alias

Do not send a Libyan IBAN as an NPT alias. Use a real enrolled NPT handle from the target environment. Example aliases in docs are placeholders, not production test fixtures.

Phone ​

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/phone/start
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>
Content-Type: application/json
json
{
  "phone": "0911091195"
}

Then verify the phone OTP:

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/phone/verify
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>
Content-Type: application/json
json
{
  "otp_code": "123456"
}

Show the returned masked account choices only, then bind the selected account:

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/phone/select
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>
Content-Type: application/json
json
{
  "account_index": 0
}

4. Select bank authorization ​

Use one of the auth modes Astro returned for that session.

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/select-auth
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>
Content-Type: application/json
json
{
  "auth_mode": "OTP"
}

For push authorization, use PUSH and poll /push-status.

5. Submit OTP or poll push ​

For OTP:

http
POST /api/v1/payments/sessions/{session_id}/merchant-checkout/confirm
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>
Content-Type: application/json
json
{
  "otp_code": "123456"
}

For push:

http
GET /api/v1/payments/sessions/{session_id}/merchant-checkout/push-status
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>

Poll every 2 to 5 seconds until the state is terminal.

6. Confirm final result ​

The customer page may show progress from:

http
GET /api/v1/payments/sessions/{session_id}/merchant-checkout/status
Origin: https://ezone.ly
X-Session-Token: <merchant_checkout_session_token>

Fulfilment must still wait for a signed final webhook such as payment.completed, or an equivalent server-side final status lookup recorded by the merchant backend.

Sandbox checklist ​

  • Create a trusted session with Idempotency-Key.
  • Exchange /merchant-checkout/access from the backend.
  • Resolve NPT alias, IBAN, and phone inputs correctly.
  • Complete OTP or push in sandbox.
  • Receive and verify a signed webhook.
  • Store merchant order reference, Astro session id, and webhook event id.

Live checklist ​

  • Use mk_live_... or a live channel key only on backend infrastructure.
  • Use a live webhook endpoint and live webhook secret.
  • Keep sandbox and live logs separate.
  • Confirm allowed browser origins include the live merchant domain.
  • Do not fulfil from browser-only state.
